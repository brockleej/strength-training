# Feature Ideas — March 28, 2026

Generated by analyzing the current state of the app, researching competitor features, and reviewing recent iOS/SwiftUI capabilities.

## Feature 1: Smart Rest Timer with Mode-Aware Defaults

### What it is

An automatic rest timer that starts counting the moment a set is logged, with intelligent default durations based on the current training mode and exercise type. The timer would appear as a compact, non-intrusive overlay at the bottom of the active workout screen, showing elapsed time and a progress ring. When the recommended rest period ends, the user gets a haptic nudge (leveraging the existing `HapticService`).

### Why it matters

Rest period management is the single most universally requested feature in strength training apps. Every major competitor (Strong, Hevy, StrongLifts, Setgraph) includes one, and for good reason: rest duration directly affects training stimulus. Resting too long during endurance work reduces metabolic stress; cutting rest short during heavy strength sets compromises recovery and the next set's performance. Currently, the app has no awareness of time between sets, so users must either watch the gym clock or use a separate timer app, which breaks their flow.

### How it would work

The timer would be mode-aware by default: Strength mode sets a 3-minute rest, Endurance mode sets a 90-second rest. Users could override per-exercise via a long-press on the timer or in the exercise library (a new `restSeconds` optional field on `Exercise`). The timer auto-starts after each call to `addSet()` in `WorkoutViewModel` and resets when the next set is logged. A small `RestTimerView` component would sit above the set input area in `SetInputView`, using a circular progress indicator with the remaining time in the center. Tapping adds/removes 15 seconds (matching industry convention). The `completedAt` timestamp already exists on `SetRecord`, so no schema migration is needed to compute rest durations retroactively for existing data. Over time, average rest durations per exercise could feed into the Progress tab as a secondary metric.

### Implementation scope

- New `RestTimerViewModel` (manages countdown, haptic trigger)
- New `RestTimerView` (compact circular progress + time label)
- Small addition to `Exercise` model: optional `restSeconds: Int?`
- Hook into `addSet()` in `WorkoutViewModel` to start/reset timer
- Estimated effort: ~2–3 days

## Feature 2: Progressive Overload Suggestions ("Next Set Target")

### What it is

Before the user starts logging sets for an exercise, the app would display a "Target" card showing a recommended weight and rep scheme for today's session. The suggestion is computed locally from the user's own historical data using a simple, transparent algorithm — not a black-box AI model.

### Why it matters

The core purpose of a strength training tracker is to facilitate progressive overload: doing slightly more over time. Right now the app shows what the user did *last time* (via `LastSessionBanner`), but it doesn't help them decide what to do *this time*. That mental math — "I did 135 lbs × 8 last week, should I try 140 × 8 or 135 × 9?" — is the single most important decision in every workout, and it's the decision the app currently leaves entirely to the user. Competitors like Dr. Muscle, Fitbod, and Jefit have moved aggressively into auto-regulated suggestions, and users increasingly expect their tracker to be a *coach*, not just a logbook.

### How it would work

The algorithm would be deliberately simple and explainable. For each exercise + training mode combination, the app looks at the user's last 3–4 sessions and applies one of two micro-progression rules:

**Strength mode:** If the user hit their target reps (e.g., 5) on all working sets last session → suggest increasing weight by 5 lbs at the same reps. If they didn't hit target reps on every set → suggest the same weight, aiming for one more total rep than last time.

**Endurance mode:** If the user completed 12+ reps on all working sets → suggest increasing weight by 5 lbs at 10 reps. Otherwise → suggest same weight, targeting one more rep per set.

The suggestion would appear as a subtle card inside the expanded `ExerciseRowView`, just below the `LastSessionBanner`, styled as a forward-looking complement to it (e.g., "Last: 135 × 8 → Target: 140 × 8" or "Last: 135 × 6 → Target: 135 × 7"). The user can dismiss or ignore it — it's guidance, not a mandate.

A new lightweight `ProgressionService` would house the algorithm. It reads from existing `SetRecord` data via `ModelContext` queries, needs no new model fields, and could later be extended with user-configurable progression increments (2.5 lb micro-plates, percentage-based progression, etc.). The Progress tab could eventually track "target hit rate" — how often the user meets or exceeds the suggestion — as a motivational metric.

### Implementation scope

- New `ProgressionService` (pure logic, testable, ~100 lines)
- New `TargetSuggestionView` (small card component)
- Integration point in `ExerciseRowView` expanded content
- No model changes required
- Estimated effort: ~2–3 days

## Improvement: Adopt Liquid Glass for the Workout Tab Navigation

### What it is

Replace the current flat, opaque day-type picker cards and training mode toggle on the workout screen with iOS 26's Liquid Glass material, giving the app a modern, platform-native feel that aligns with the system design language.

### Why it addresses a real issue

The git history shows a launch screen fix that was attempted but noted as "still not working yet" (`e4e40a8`), and there's a known workaround for the share sheet rendering blank inside a `.sheet` modal. These are symptomatic of the app not yet being fully refreshed for iOS 26's design expectations. More importantly, iOS 26 introduced Liquid Glass as the *default* navigation-layer material, and apps that don't adopt it are starting to look dated next to updated system apps. The `WorkoutDayPickerView` with its three color-coded cards and the `TrainingModePickerView` toggle are the most prominent "landing" surfaces in the app — refreshing them would have an outsized visual impact for relatively little code.

### How it would work

The `WorkoutDayPickerView` cards would be wrapped in a `GlassEffectContainer` with each card applying `.glassEffect()` instead of the current `.background(dayType.color.opacity(...))` fill. The day-type color would shift from being the card background to being an accent tint on the icon and text, letting the translucent glass material show the scroll content beneath. Similarly, the `TrainingModePickerView` segmented toggle would use `.glassEffect()` for its background capsule, replacing the current manual shadow/border/opacity styling (which has multiple hardcoded values like `0.12` opacity, radius `2`, offset `y:1`). This also addresses the code-quality concern of hardcoded styling constants scattered across views — the glass material handles highlight, shadow, and depth automatically.

The launch screen issue could also be addressed in the same pass by switching from the auto-generated launch screen (`INFOPLIST_KEY_UILaunchScreen_Generation = YES`) to a minimal custom `LaunchScreen.storyboard` with the app icon centered on a solid background, which is more reliable across iOS versions.

### Implementation scope

- Update `WorkoutDayPickerView` to use `GlassEffectContainer` + `.glassEffect()`
- Update `TrainingModePickerView` to use `.glassEffect()` background
- Replace auto-generated launch screen with a simple storyboard
- Remove hardcoded shadow/border/opacity values that glass replaces
- Estimated effort: ~1 day
